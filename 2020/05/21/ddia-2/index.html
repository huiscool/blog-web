DDIA 笔记 (2) 数据模型和查询语言

DDIA
发布日期:   2020-05-21
文章字数:   1.1k
阅读时长:   4 分

关系模型vs文档模型 Relational Model Versus Document Model

尽管关系模型得到广泛的应用，但与编程所用的面向对象仍存在一定的差别。需要对象关系映射ORM作为中间层。

而对于文档模型，比如 json，它比多表的数据有更好的局部性locality，但是不利于关联性的查询。

注意的是，即使应用的初期没有join关联的需求，后期也有可能出现关联的需求。数据总是趋于相互联系的：

Data has a tendency of becoming more interconnected as features are added for applications.

层次模型（hierarchical model ）很适合处理一对多的关系，但是对于多对多的关系显得无能为力，并不支持join操作。解决这个问题导致了关系模型和网络模型的诞生。

网络模型

网络模型与层次模型的不同点在于每个节点可以有多个父节点，因此支持了多对一和多对多的关系，这些链接不是关系模型中的外键，而是类似于持久化了的指针。程序员为了查一条数据，需要从根节点出发，找到相应的查询路径。

网络模型的缺点：

  • 查询语句不好写
  • 数据模型不好改

关系模型和文档数据库

关系模型并不适合于所有场景，比如对于树状的数据，使用文档结构存储就比把它拆分成一张张表好。文档模型有很好的模式灵活性。

schemeless: 任意键值对都可以加入到文档中，读者对读到的模式没有任何保证

schema-on-read: 读入时假定文档已经有相应的schema（隐式），读取时进行验证。一般指关系数据库中存的json/xml等结构化文档

schema-on-write:写入时已经有相应的模式，数据库已经保证写入的数据满足相应的模式。一般指常见的关系型数据库。

文档模型中的局部性

查询时，文档模型的局部性仅适用于大部分读出来的场景，比如模板渲染。只读少量字段的场景和修改场景都不太适合。而它的修改并不方便，大部分时候需要覆写原来的文档。

查询语言 Query Languages for Data

描述型语言和命令型语言

描述型语言的好处：更简洁易读；与物理结构无关；便于优化器进行通用的优化。

MapReduce Querying

MapReduce 既不是描述型也不是命令型，它仅仅提供了一个low-level的API。有些时候为了支持复杂的查询，这些NoSQL数据库是在 “reinvent SQL”。

图状数据模型 Graph-Like Data Models

图状数据模型： 将异质的数据放在同一张表里

属性图 Property Graphs

属性图可以表达一些传统关系模型不易表达的数据，比如不同国家之间的行政区划。不同国家会分不同的级别，但是用图模型可以保持数据模型的清晰。

图模型更容易适应应用数据结构的更改。

Cypher Query Language

一种用在属性图上的声明式语言。

利用SQL执行图式查询 Graph Queries in SQL

区别： 传统查询你可以预先知道join的次数，也就是需要多少实体之间的联系；然而在图模型中，为了找一个实体，需要经历多条边，因此join的次数预先不知道。

实现：递归公用表表达式 (recursive common table expressions, CTE, 在SQL Server等特定数据库才支持）

三元组存储 Triple-Stores and SPARQL

三元组：（主语subject，谓语predicate，宾语object） 例子：（Jim, likes, banana）

语义网络 The semantic web

希望把互联网上的文本变成一个机器可读的分布式数据库。

RDF data model

RDF 与三元组存储的区别：

  • RDF用的是XML;
  • RDF采用中的主谓宾都是URL，把域名当做一个命名空间；

SPARQL

RDF上的查询语言，类似cypher.
