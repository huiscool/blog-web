DDIA 笔记 (7) 事务

DDIA
发布日期:   2020-07-19
文章字数:   1.8k
阅读时长:   6 分

事务

ACID的理解

  • Atomic – all or nothing

    • 并发中提到的原子操作：一个线程执行的原子操作，另一个线程只看到它操作之前或操作之后的状态，而看不到中间状态；
    • 事务中的原子性：与事务的并发无关，而是多个写操作的事务执行过程中如果被打断（断电），数据库仅处于事务之前或事务之后的状态。

  • Consistency

    • 一致性有多种理解：
      • 在分布式系统中的副本一致性指不同副本具有相同的值；
      • 一致性哈希指某些系统用于重新分区的一种方法；
      • CAP理论中，一致性指可定序（可线性化）；
    • 事务中的一致性：指数据库在应用程序的特定概念中处于“良好状态”，也就是对数据的一组约束必须始终成立。它实质上是应用层来保证的。

  • Isolation

    • 同时执行的事务是相互隔离的，它们彼此不能相互影响。

    • 数据库确保当事务已经提交时，结果与它们按顺序运行（一个接一个）是一样的，尽管实际上它们可能是并发运行的

    • 数据库提供不同的隔离等级来保证隔离性

  • Durability

    • 持久性 是一个承诺，即一旦事务成功完成，即使发生硬件故障或数据库崩溃，写入的任何数据也不会丢失。
    • 单机数据库，意味着数据已经写入存储介质；
    • 分布式带复制的数据库，意味着数据已经复制成功；

问题：单个对象（数据库中的某一行）的并发写是事务吗？答案是否定的。

事务通常被理解为，将多个对象上的多个操作合并为一个执行单元的机制。

（和操作系统常见的锁不同，我们必须关注多对象之间的关系和可能出现的差错）

怎样确定是同一个事务里面的操作？一般的关系型数据库采用同一条TCP连接来区分。

弱隔离级别

问题：脏读、脏写、读取偏差（不可重复读）、更新丢失、写偏差、幻读

脏读 – 事务A读到事务B未提交的写入

脏写：事务A事务B几乎同时写入对象a。事务B如果可以覆盖事务A已写入但未提交的内容，称为脏写

解决脏读、脏写的一个隔离等级称为read commented.

解决脏写：

  • 给每一行增加一个行锁，事务想要修改某一行时，需要先获得行锁；同一个事务可能同时会有多个行锁。如果另一个事务想修改同一行，那它需要等到第一个事务提交或者中止后，才能获得该锁。

解决脏读：

  • 读锁，但会导致读取事务等待一个慢写入的事务

  • 获得行锁的事务记一个新值，其他读取该行的事务用旧值

读取偏差，不可重复读：同一个事务之中两次读取之间的数据库状态不同

解决读取偏差：

  • 快照隔离，数据库在一个对象中维护多个时间点上的值

  • 事务具有一个自增ID，这个ID相当于时间点；事务只能读取到时间点比他早的值

更新丢失：采用读取-修改-写入方法的并发事务，可能会导致前一个事务失效

解决方法：

  • 原子写入：update counters SET value = value + 1 where key = 'foo'

  • 锁定某行，让对某个对象的读取延迟到某次写入之后 (for update 语句)

  • Compare And Set (CAS)

    • UPDATE wiki_pages SET content = '新内容' WHERE id = 1234 AND content = '旧内容';

  • 是否安全依赖于数据库快照隔离的实现；

  • 数据库本身自动检测（MySQL不检测）

题外话：分布式数据库中如果采用”最后写入胜利LWW”的冲突解决方案，则很有可能出现更新丢失的情况。

写入偏差：两个事物读取相同的对象集合，然后更新其中的一些对象。如果更新的对象不同，可能会发生写入偏差；如果更新的对象相同，则可能发生脏写或丢失更新

写入偏差的模式

  • 先SELECT查出一堆行；
  • 代码判断是否满足一定条件，不满足会放弃事务；
  • 执行一个会改变查询先决条件的写入，并提交事务。

解决方法：

  • for update 锁定一部分行，并告知数据库这些行用于更新；

    • 这种方式仅适用于查询时能返回一部分行并修改这些行的情况。如果是先查询，发现没有，然后再添加一行，使得查询先决条件改变，那我们就不能预先锁定不存在的行

  • 触发器？

  • 物化冲突

    • 把多个对象的冲突变成具体的行上的冲突
    • 不到万不得已不要用

  • 可序列化Serializable的隔离等级

可序列化

  • 实现
    • 单线程上的串行 real serialized
    • 两相锁定
    • 乐观并发控制技术

单线程上的串行

  • 兴起原因

    • RAM的降价，我们可以将足够多的数据放到内存中；
    • OLTP的事务很短，一般只进行少量的读写，分析事务一般只读，用快照隔离即可

  • 常用于内存数据库(Redis)

  • 免去锁的开销

  • 吞吐量取决于单核的吞吐量

  • 需要避免像传统数据库的事务让应用程序和数据库之间交互，因为不能让数据库和应用程序之间走网络连接（否则网络之间通信需要花费大量时间，让事务吞吐量大为下降）

    • 把事务放到存储过程中（Redis用Lua写存储过程）

  • 分区时需要尽可能避免跨分区锁定，吞吐量太低

两阶段锁定（不是两阶段提交）

  • 和操作系统中的读写锁极像，第一阶段在事务执行时获得锁，第二阶段在事务结束时释放所有的锁

    • 如果事务A读取了一个对象，并且事务B想要写入该对象，那么B必须等到A提交或中止才能继续。 （这确保B不能在A底下意外地改变对象。）

    • 如果事务A写入了一个对象，并且事务B想要读取该对象，则B必须等到A提交或中止才能继续。 （读取旧版本的对象在2PL下是不可接受的。）

  • 性能是个大问题
    • 死锁可能性大大增加；
    • 拖慢其他包含对象的查询和事务
  • 谓词锁 对满足特定条件的对象进行锁定，可以用于新增数据的锁定；
  • 索引范围锁：根据谓词锁定索引上一段连续对象

序列化快照隔离

  • 怎样知道有并发：
    • 检测对旧MVCC对象版本的读取（读之前存在未提交的写入）
    • 检测影响先前读取的写入（读之后发生写入）
  • 有并发后咋做：
    • 提交时abort，让应用程序重试
